# 第0章 设计哲学

## 0.1 核心设计原则

X 语言的设计围绕以下核心理念展开：

### 可读性第一
```
Principle: Code is read more often than it is written.
```

**设计选择：**
- 使用自然语言风格的关键字（`needs`、`given`、`wait`、`when`/`is`）
- 语法尽量接近人类思维方式
- 避免过度符号化的晦涩语法
- 鼓励清晰的表达而非技巧性的简写

---

### 显式优于隐式
```
Principle: Make important things visible.
```

**设计选择：**
- 效果必须显式声明（`throws`、`async`、`io`）
- 变量可变性需要 `mut` 关键字
- 导入路径清晰明确
- 类型注解可选但推荐
- 重写需要 `override` 关键字

---

### 安全与性能兼得
```
Principle: Safety without sacrificing performance.
```

**设计选择：**
- Perceus 编译时引用计数，无 GC 停顿
- 所有权系统防止内存错误
- 代数数据类型与穷尽匹配
- 零成本抽象
- 编译期计算支持

---

### 多范式支持
```
Principle: Use the right paradigm for the problem.
```

**设计选择：**
- 函数式：纯函数、不变数据、高阶函数
- 命令式：可变状态、循环、顺序执行
- 面向对象：类、继承、多态
- 声明式：模式匹配、表达式导向

---

### 实用主义
```
Principle: Perfect is the enemy of good.
```

**设计选择：**
- 提供逃逸 hatch（unsafe、FFI）
- 允许渐进式类型化
- 支持与现有生态系统互操作
- 允许局部的状态和副作用

---

## 0.2 语言特性谱系

### 关键特性设计决策

| 特性 | 设计决策 | 理由 |
|------|----------|------|
| **变量声明** | `let` / `let mut` | 显式可变性，Rust 风格 |
| **注释** | `//` / `/** */` | 熟悉的语法，工具支持好 |
| **类型系统** | 结构化类型 + 名义类型 | 灵活且安全 |
| **内存管理** | Perceus (ARC + 重用分析) | 无 GC，低开销，线程安全 |
| **效果系统** | R·E·A 显式效果 | 副作用可见，可组合 |
| **错误处理** | Result 类型 + 异常 | 可选的恢复策略 |
| **并发** | async/await + together/race | 结构化并发 |
| **模式匹配** | when/is 表达式 | 强大的解构能力 |

---

## 0.3 命名与风格

### 关键字设计
```
Keyword Design Space:
  - 自然语言风格: needs, given, wait, when, is
  - 传统编程风格: let, fun, if, else, for, while
  - 特殊值: true, false, null, none, some, ok, err
```

**设计理念：**
- 控制流使用熟悉的传统关键字
- 新概念使用自然语言风格关键字
- 避免过度创新，降低学习门槛

---

## 0.4 与其他语言的关系

### 灵感来源

| 语言 | 借鉴的特性 |
|------|-----------|
| **Rust** | 所有权、模式匹配、错误处理、显式可变性 |
| **ML/Haskell** | 代数数据类型、类型推断、高阶函数 |
| **Swift** | 可选类型、结果类型、灵活的枚举 |
| **Kotlin** | 表达式导向、扩展函数、空安全 |
| **Zig** | 编译期计算、无 hidden control flow |
| **Lean** | 效果系统的设计思路 |

### X 语言的独特之处
- **Perceus 内存管理**：编译时 ARC + 重用分析，无 GC
- **R·E·A 效果系统**：显式、可组合的副作用追踪
- **自然语言关键字**：`needs`、`given`、`wait`、`when`/`is`
- **结构化并发**：`together`、`race`、`timeout`
- **原子性**：`atomic` 块 + `retry` 的 STM 风格

## 0.5 非目标

X 语言**不**追求：

1. **成为一切语言**：专注于系统编程、应用开发、工具编写
2. **零学习成本**：好的设计需要一定学习投入
3. **完全向后兼容**：愿意在必要时打破兼容性
4. **与 C++ 完全兼容**：提供 FFI，但不追求 ABI 兼容
5. **在每个基准上最快**：追求足够快，同时保持安全性和生产力

---

## 0.6 演进原则

### 语言演进
```
Stability > Experimentation > Deprecation > Removal
```

**策略：**
- 新特性通过 Preview 标志 gated
- 收集反馈后稳定化
- 缓慢废弃，长期支持
- 重要变更有迁移路径
